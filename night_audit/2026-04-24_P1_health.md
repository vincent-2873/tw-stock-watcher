## C.2 線上端點壓力 / 飢餓現象 ⚠️ **這是本輪最重要的隱性問題**

### 凌晨 02:00 vs 02:38 對照

| 端點 | 02:00 表現 | 02:38 表現 | 推論 |
|---|---|---|---|
| `GET /api/time/now` | ❌ 30s timeout, 0 byte | ✅ 1.67s | 暫時性 hang |
| `GET /api/chat/health` | ❌ 20s timeout, 0 byte | ✅ 0.15s | 暫時性 hang |
| `GET /health` | ❌ timeout | ✅ 0.68s | 暫時性 hang |
| `GET /api/market/overview` | ✅ 200 OK ~5s | ✅ 同樣 OK | 一直可用 |
| `POST /api/chat`(SSE) | ✅ 正常 streaming | ✅ 正常 | 一直可用 |
| Frontend `/` | ✅ 200 OK 0.4s | ✅ 同樣 OK | 一直可用 |

### 根因分析(讀 code 後推論)

**`/api/time/now` 是同步函式**(`def time_now()`,non-async),FastAPI 會在 thread pool 執行。
**`/api/chat/health` 雖是 async,但呼叫 `resolver_stats()` → `_ensure_loaded()` 會嘗試取 `threading.Lock`**,如果有人正在 reload 持鎖,就會卡 event loop。

當有長時間 chat streaming 在跑(每個 8-12s),uvicorn single worker 的 thread pool 跟 event loop 都被佔用 → 輕端點被「飢餓」。

### 影響

🟡 **Vincent 看到的「網站打不開」現象的真兇**:首頁載入時要打 `/api/time/now`(Hero 權威時鐘)+ `/api/quack/picks` + 其他幾個輕端點。如果剛好遇到飢餓窗口,這些 fetch promise 卡住 → React 等不到資料 → 整頁空白。

🟢 **重端點(對話 AI / market/overview)反而沒事**,因為 async + 走自己的 task。

### 目前狀態

✅ **02:38 起恢復健康**,但**這個 architecture 隱患還在**,負載一上去就會復發。

### 建議(列 3 個方案,等 Vincent 決定)

#### 方案 1(最快 5 分鐘):把 `/api/time/now` 改 `async def`

```python
@router.get("/time/now")
async def time_now():   # 加 async
    ...
```

純函式沒打外部 API,改 async 零風險,直接走 event loop 不再排隊 thread pool。

#### 方案 2(5 分鐘):`chat_health` 不要呼叫 `resolver_stats()`

讓 health 端點純粹回 `{ok, model, tpe_now}` 不要碰 resolver。Resolver 狀態另開一個專門端點。

#### 方案 3(複雜):換 uvicorn → gunicorn + 多 worker

每個 worker 各自有 event loop / thread pool,不互相飢餓。但會增加記憶體成本。

→ **建議組合 1+2,最便宜最有效**。
