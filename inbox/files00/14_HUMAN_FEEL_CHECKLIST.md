# 👤 呱呱招待所 — 人味檢查手冊（Human-Feel Checklist）

> **這份文件是給 Claude Code 的最後一道品質關卡**。
>
> 現有設計文件告訴 Claude Code「要做什麼」，這份文件告訴 Claude Code 「做完後怎麼檢查是否讓人用起來舒服」。
>
> **這是解決「Claude Code 操作給人不人性」問題的核心文件。**

---

## ⚠️ 為什麼需要這份文件？

Vincent 給 Claude Code 的回饋：
> 「操作關於人的都可以大讚能力，但操作給人還是有點難。」

這句話點破一個關鍵問題：

| Claude Code 擅長 | Claude Code 不擅長 |
|---|---|
| 分析使用者需求（抽象思考）| 感受「被使用者操作」的體驗 |
| 寫出功能正確的程式碼 | 寫出用起來「舒服」的程式碼 |
| 實作元件的外觀 | 拿捏元件的「呼吸感、節奏感」 |
| 接 API 給資料 | 處理「等待、loading、空狀態」的使用者情緒 |

**所以，這份文件專門補上這個缺口**。

---

## 📋 10 大人味檢查面向

每次實作完一個頁面或元件，Claude Code **必須逐條檢查**。如果有 ❌，要**回去改**，不要交給 Vincent。

### 檢查面向總覽

1. 🎯 **方向感**：使用者知道自己在哪裡嗎？
2. ⏱️ **等待感**：使用者等待時感受如何？
3. 🫱 **回饋感**：使用者點擊後有感覺嗎？
4. 🎼 **節奏感**：頁面整體律動舒服嗎？
5. 🌬️ **呼吸感**：頁面留白夠嗎？
6. 🫰 **拇指感**：手機單手能用嗎？
7. 👀 **視線感**：視線動線順暢嗎？
8. 💭 **意圖感**：使用者下一步該做什麼清楚嗎？
9. 🎭 **情緒感**：成功/失敗的情緒傳達夠嗎？
10. 🧘 **疲勞感**：使用 5 分鐘後眼睛會累嗎？

## 🎯 面向 1：方向感（Sense of Place）

### 問題：Claude Code 常忘記告訴使用者「你現在在哪」

#### ✅ 必須檢查

**1.1 頁面有清楚的標題（H1）**
- 不是「Dashboard」這種泛泛的詞
- 要用呱呱招待所的語彙：「今日」「池塘」「筆記」
```
❌ 「股票詳情頁」
✅ 「📓 我的筆記 / 富喬 1815」（帶路徑）
```

**1.2 麵包屑（Breadcrumb）永遠存在**
- 超過一層的頁面必須有
- 點擊可以回上一層
```
✅ 池塘 > CCL 漲價 > 富喬 1815
```

**1.3 目前位置的視覺提示**
- 側邊欄當前頁面要有高亮
- 顏色用 `--accent-gold`，不要用預設藍色
- 有個小小的 3px 暗金色指示條

**1.4 URL 要有意義**
```
❌ /page?id=123
✅ /pond/ccl-price-increase
✅ /journal/6182
```

#### 🔍 測試方法
隨機打開一個內頁，5 秒內能回答這三個問題：
- 我現在在哪？
- 這頁在做什麼？
- 怎麼回去？

## ⏱️ 面向 2：等待感（Sense of Waiting）

### 問題：Claude Code 常把「Loading...」當作等待狀態就打發了

#### ✅ 必須檢查

**2.1 等待時間 vs 回饋方式**

| 等待時間 | 應該做什麼 |
|---------|-----------|
| < 100ms | 什麼都不用做（人類感知不到）|
| 100ms - 1s | 簡單的 opacity 漸變 |
| 1s - 3s | Skeleton 骨架屏 |
| 3s - 10s | 骨架屏 + 進度條或呱呱「思考中...」動畫 |
| > 10s | 呱呱主動安撫「呱呱正在仔細看，再等一下」|

**2.2 絕對不能出現的等待**
- ❌ 整個畫面白屏
- ❌ 只有一個轉圈圈 icon 在畫面中央
- ❌ 「Loading...」純文字
- ❌ 畫面跳動（內容載入後才有位置）

**2.3 Skeleton 骨架屏規格**
```
每個卡片型元件都要有對應的 Skeleton
顏色：比背景深 5%（若深色）或淺 5%（若淺色）
動畫：慢慢的 pulse（2s 週期，不要太快）
```

**2.4 呱呱介入等待時刻**
```
AI 分析（3-10 秒）：
  → 顯示【研究呱呱】+ 「呱呱正在翻資料...」
  → 每 3 秒換一句話：
     「呱呱：翻到 CCL 供應鏈了...」
     「呱呱：對照外資動態...」
     「呱呱：快好了...」
```

#### 🔍 測試方法
故意把網路調慢（Chrome DevTools → Network → Slow 3G），打開每個頁面：
- 完全沒有白屏超過 0.5 秒？
- Skeleton 跟真實內容的位置一樣？
- 等待時不會無聊？

## 🫱 面向 3：回饋感（Sense of Feedback）

### 問題：Claude Code 常做出「點了沒反應」的按鈕

#### ✅ 必須檢查

**3.1 所有可點擊元素都要有 4 種狀態**

| 狀態 | 樣式 |
|------|------|
| 預設 | 正常樣式 |
| Hover（滑鼠移上）| 背景變淺、字色加深、或微微上浮 |
| Active（按下去瞬間）| 縮小 1-2% 或顏色更深 |
| Focus（鍵盤 Tab 到）| 明顯的 outline |
| Disabled | 透明度 50% + cursor 變 not-allowed |

**3.2 按鈕點擊後的即時回饋**
```
點擊按鈕 → 以下必須在 100ms 內發生：
- 視覺：按鈕輕微按下感
- 微觸覺（手機）：navigator.vibrate(10)（可選）
- 如果有後續動作：立即顯示 loading 狀態
```

**3.3 Toast / Notification 規則**
```
操作成功：
  ✅ 右上角 Toast
  ✅ 綠色暗調（用 --fall-light 系列）
  ✅ 2 秒後自動消失
  ✅ 文案要有人味：「呱呱記下了 ✓」而非「Success」

操作失敗：
  ❌ 右上角 Toast
  ❌ 紅色暗調
  ❌ 停留 4 秒或手動關閉
  ❌ 給解法：「網路有點慢，再試一次？」+ [重試按鈕]
```

**3.4 絕對不能出現的無回饋**
- ❌ 點了按鈕沒變化
- ❌ 表單送出後不知道有沒有成功
- ❌ 加入自選股後沒視覺變化
- ❌ 複製文字後沒「已複製」提示

#### 🔍 測試方法
每個按鈕、連結、可點擊卡片：
- hover 有變化？
- 點擊瞬間有視覺回饋？
- 操作後有明確結果提示？

## 🎼 面向 4：節奏感（Sense of Rhythm）

### 問題：Claude Code 常把所有內容塞成一坨，沒有節奏

#### ✅ 必須檢查

**4.1 垂直節奏（Vertical Rhythm）**

頁面由上到下要有清楚的視覺節奏：
```
大標題 ─ 32px
  │ ← 24px 間距
摘要文字 ─ 16px
  │ ← 16px 間距
小標題 ─ 20px
  │ ← 16px 間距
內文 ─ 14px
  │ ← 32px 間距（區塊間大間距）
下一個大標題 ─ 32px
```

**4.2 留白原則（侘寂風核心）**
- 每個卡片的 padding 至少 24px
- 卡片之間至少 16px 間距
- 頁面兩側至少 24px margin（手機）或 80px（電腦）
- **大區塊之間至少 48px 空白**

**4.3 文字密度限制**
```
❌ 每行超過 80 字
✅ 每行 40-60 字（中文則是 20-30 字）

❌ 一段超過 5 行
✅ 每段 2-3 行

❌ 一頁沒有小標
✅ 每 4-5 段加一個小標
```

**4.4 卡片內部節奏**
```
卡片上方 → 標題 + icon（32px 高）
        ─── 16px 間距
       → 主內容（彈性）
        ─── 16px 間距
        → 底部 CTA（36px 高）

padding: 24px
總高度：約 140-200px（不要超過螢幕一半）
```

#### 🔍 測試方法
瞇眼看頁面，感覺像：
- 🎵 有起伏的音樂？✅
- 📏 平均分布的數字？❌
- 🗑️ 塞滿資訊的垃圾場？❌❌

## 🌬️ 面向 5：呼吸感（Sense of Breathing）

### 問題：Claude Code 常把首頁當「我要把全部功能都塞上去」

#### ✅ 必須檢查

**5.1 首頁資訊量上限**
- ✅ 5-6 個核心區塊（已在 UX 設計中）
- ❌ 不要超過 7 個主要區塊
- 捨棄原則：**任何「順便放上去」的東西都要刪掉**

**5.2 每個區塊的上限**
- 題材卡 Top 5（不是 Top 10）
- 自選股顯示 3-4 檔（不是全部）
- 新聞 3-5 則（不是 20 則）

**5.3 留白佔比檢查**
```
首頁截圖放大看，計算：
- 內容（文字、卡片、圖）佔比
- 留白佔比

目標：留白佔 35-40%
❌ 內容佔比 > 70% = 太擠
❌ 留白佔比 > 60% = 太空
```

**5.4 圖示（Icon）使用原則**
- 每個主區塊一個 emoji/icon 就好
- ❌ 每行文字前面都加 emoji
- ❌ 每個欄位都有小圖示
- 記住：**侘寂風是「少」不是「多」**

#### 🔍 測試方法
把首頁截圖給朋友看 3 秒然後遮住：
- 記得看到哪些東西？
- 超過 5 個 → Claude Code 做錯了
- 剛好 4-5 個 → 完美
- 少於 3 個 → 可能太空

## 🫰 面向 6：拇指感（Sense of Thumb Reach）

### 問題：Claude Code 常做出「只能用兩隻手的手機版」

#### ✅ 必須檢查

**6.1 手機版主要按鈕必須在「拇指舒適區」**

```
┌─────────────────┐
│ ❌ 危險區       │  ← 頂部很難按
│                  │
├─────────────────┤
│                  │
│ ⚠️ 勉強可及     │  ← 中間要伸
│                  │
├─────────────────┤
│                  │
│ ✅ 舒適區       │  ← 拇指天然落點
│                  │
├─────────────────┤
│ ✅ 舒適區       │  ← 最好的位置
└─────────────────┘
```

**關鍵元件位置**：
- ✅ 底部 Tab Bar（主導航）
- ✅ 浮動呱呱（右下角，60px × 60px）
- ✅ 主要 CTA 按鈕（在卡片底部，不是頂部）
- ✅ 返回按鈕（頂部左側，可用下拉手勢輔助）

**6.2 按鈕最小觸控區**
- iOS Human Interface: 至少 44 × 44 px
- Material Design: 至少 48 × 48 dp
- **呱呱招待所標準：至少 48 × 48 px**

**6.3 相鄰按鈕的間距**
- 兩個按鈕之間至少 8px（避免誤觸）
- 重要/危險按鈕要更遠（12-16px）

**6.4 手勢支援**
- 左右滑動：切換題材卡、個股卡（carousel）
- 下拉：重新整理（關鍵頁面）
- 長按：快速選單（加入自選、設警示）

#### 🔍 測試方法
單手拿手機，拇指不碰到螢幕上半部：
- 能完成 90% 的操作？✅
- 需要換手才能點某些按鈕？❌

## 👀 面向 7：視線感（Sense of Eye Flow）

### 問題：Claude Code 常把頁面做成「要到處找」的大賣場

#### ✅ 必須檢查

**7.1 Z 字形閱讀模式（電腦版）**

人類閱讀自然路徑：
```
左上 ─────→ 右上
             ↙
          ↙
左下 ─────→ 右下
```

**關鍵內容放在 Z 的轉角**：
- 左上：Logo + 頁面標題
- 右上：使用者資訊 / 通知
- 中間：主要內容（呱呱功課、題材）
- 左下：次要資訊
- 右下：浮動呱呱 / CTA

**7.2 手機版垂直掃描模式**

手機拇指滑動方向就是視線方向：
```
第 1 屏（不滑動）→ 最重要（呱呱功課）
第 2 屏（滑一下）→ 重要（題材熱度）
第 3 屏（滑兩下）→ 次要（自選股、警示）
```

**絕對不能**：把「今日重點」放在第 3 屏要捲動才看得到。

**7.3 視覺重量（Visual Weight）順序**

同一個區塊中，視覺重量由大到小：
```
1. 主標題（24-32px，粗體）
2. 關鍵數字（股價、熱度，大字）
3. 次標題（16-18px）
4. 內文（14-16px）
5. 輔助資訊（12px，低對比）
```

**7.4 色彩焦點**
- 一頁只能有 1-2 個「真正吸引眼球」的焦點
- 焦點用 `--accent-gold` 或熱度色
- 其他都用灰階

#### 🔍 測試方法
5 秒內給朋友看頁面：
- 第一眼看到什麼？
- 是你想讓他看到的嗎？

## 💭 面向 8：意圖感（Sense of Intent）

### 問題：Claude Code 常把使用者丟在頁面上，不告訴他下一步

#### ✅ 必須檢查

**8.1 每個頁面都要有「下一步建議」**

```
首頁：
  ↓ 下一步：點題材卡 → 進題材詳情頁

題材詳情頁：
  ↓ 下一步：點個股 → 進個股詳情

個股詳情頁：
  ↓ 下一步：加入自選 or 設警示 or 問呱呱
```

**不要**：讓使用者點完某個功能後不知道要幹嘛。

**8.2 空狀態（Empty State）設計**

❌ **爛的空狀態**：
```
你還沒有自選股
```

✅ **好的空狀態**：
```
🦆 [睡覺呱呱]
還沒有自選股

呱呱建議你從這裡開始：
[ 瀏覽熱門題材 → ]
[ 看呱呱今日推薦 → ]
```

**8.3 完成操作後的引導**

❌ **爛的引導**：操作成功後，停在原頁
✅ **好的引導**：
```
✅ 已加入自選股
「呱呱記下了。要順便設個警示嗎？」
[ 設警示 ] [ 不用 ]
```

**8.4 首次使用引導（Onboarding）**

第一次使用的使用者：
- 呱呱從右下角探頭：「呱呱 👋 第一次來？」
- 5 個步驟導覽：天氣卡、功課卡、題材、筆記、浮動呱呱
- 任何一步可跳過
- 完成後不再出現

#### 🔍 測試方法
讓沒用過的人用 3 分鐘：
- 他會主動探索？✅
- 他卡在某一頁不知道幹嘛？❌

## 🎭 面向 9：情緒感（Sense of Emotion）

### 問題：Claude Code 常忘記系統要「有情緒」

這是呱呱招待所**最核心**的差異化，Claude Code 最容易做不到的部分。

#### ✅ 必須檢查

**9.1 成功的情緒**

```
✅ 加入自選 → 呱呱微笑點頭 + 小音效（選配）
✅ 達到第一目標 → 呱呱舉杯慶祝
✅ R:R 比超過 1:3 → 呱呱：「這筆漂亮！」
✅ 連續獲利 3 筆 → 呱呱送個小驚喜
```

**9.2 失敗的情緒**

```
❌ 觸發停損 → 呱呱不說教，只遞上一杯茶：
   「先休息一下，記錄下這筆。」

❌ 連續虧損 → 呱呱主動：
   「最近市場有點躁，要不要休息一天？」
   [ 看今日功課 ] [ 今天不看盤 ]
```

**9.3 中性的情緒**

```
🌊 盤整日（大盤 ±0.5%）→ 呱呱打禪定：
   「今天的池塘很安靜，
    適合整理筆記。」
   [ 看筆記 ] [ 研究新題材 ]

🌙 晚上 10 點後 → 呱呱睡覺：
   「呱呱要休息了。
    明天再一起看盤。」
```

**9.4 絕對不能出現的情緒**

- ❌ 使用者虧損時跳出「恭喜您賺錢」的廣告
- ❌ 使用者正在研究時強制彈出通知
- ❌ 使用者失敗時用紅色大字說「錯誤」
- ❌ 用「!!!」或全大寫來製造情緒

#### 🔍 測試方法
連續使用系統 1 週：
- 有一次覺得「呱呱懂我」？✅
- 有一次覺得系統很冷冰冰？❌

---

## 🧘 面向 10：疲勞感（Sense of Fatigue）

### 問題：Claude Code 常用高對比、強色來「抓眼球」，但看久了眼睛會累

#### ✅ 必須檢查

**10.1 對比度控制**

```
文字對比度（WCAG 標準）：
- 正常文字：至少 4.5:1
- 大文字：至少 3:1
- 但不要超過 15:1（太累）

✅ 主文字：#E8E3D5 on #1A1815（約 12:1，舒服）
❌ 純白 on 純黑（21:1，過累）
```

**10.2 顏色飽和度**

侘寂風配色都是**低飽和度**：
```
✅ --rise: #C9754D（暗紅銅）
✅ --fall: #6B8E5A（暗苔綠）
❌ #FF0000（純紅）
❌ #00FF00（純綠）
```

**Claude Code 如果看到規範外的高飽和色，一律改掉**。

**10.3 動畫疲勞**

```
✅ 可以有的動畫：
- Hover 時 0.2-0.3 秒的漸變
- 熱度條慢慢流動（無限循環，但慢）
- 呱呱每 5 秒微微動一下
- 數字改變時 0.5 秒的緩動

❌ 不可以的動畫：
- 閃爍（每秒超過 2 次）
- 快速旋轉
- 持續抖動（焦慮感）
- 紅燈閃爍警示（股票類最忌諱）
```

**10.4 資訊密度疲勞**

連續使用 30 分鐘後會累的設計：
- ❌ 每個角落都有通知紅點
- ❌ 時時跳出的 Toast
- ❌ 動態變化的數字（未 debounce）

舒服的設計：
- ✅ 只有重要事件通知
- ✅ 數字變化有 1-2 秒 debounce
- ✅ 允許「勿擾模式」

#### 🔍 測試方法
連續使用 30 分鐘：
- 眼睛覺得累？→ 調整對比、降低飽和
- 想暫停一下？→ 減少動畫、降低通知

## 🎯 實戰檢查清單（給 Claude Code 每完成一頁後使用）

### 📋 「做完就跑一遍」檢查表

每完成一個頁面或重要元件，**逐條確認**：

#### 視覺層面
- [ ] 配色只用侘寂風的那幾個顏色（沒有鮮豔色）
- [ ] 字體大小有在規範的 7 個階層內
- [ ] 間距有遵守 8px 倍數
- [ ] 留白佔比看起來至少 30%
- [ ] 一頁焦點不超過 2 個

#### 互動層面
- [ ] 所有按鈕有 hover / active 狀態
- [ ] 點擊後有即時視覺回饋
- [ ] Loading 狀態有 Skeleton 不是白屏
- [ ] 空狀態有引導下一步
- [ ] 錯誤狀態有明確解法

#### 動線層面
- [ ] 頁面有清楚的 H1 標題
- [ ] 超過一層有麵包屑
- [ ] 完成操作後有「下一步」建議
- [ ] 使用者不會卡在頁面不知道幹嘛

#### 手機層面
- [ ] 主要按鈕在拇指舒適區
- [ ] 觸控區至少 48 × 48 px
- [ ] 單手可操作 90% 功能
- [ ] 滑動手勢有支援

#### 情緒層面
- [ ] 成功操作有正向回饋（不只是 Toast）
- [ ] 失敗不會讓人焦慮（紅色要溫和）
- [ ] 有呱呱的存在感（至少 1 處）
- [ ] 文案有人味（不是系統自動口吻）

#### 疲勞層面
- [ ] 對比度舒服（不刺眼）
- [ ] 動畫速度不會太快（最少 0.2 秒）
- [ ] 沒有持續閃爍的東西
- [ ] 連續看 5 分鐘眼睛不累

## 🚨 紅旗警告（Red Flag）— 看到這些 Claude Code 必須停下來重做

以下症狀出現 **任何一個**，就是「做出來不人性」：

### 紅旗 1：白屏超過 0.5 秒
→ 必加 Skeleton 或 loading 動畫

### 紅旗 2：按鈕點了沒變化
→ 必加 hover + active + 點擊後 feedback

### 紅旗 3：使用者問「我現在在哪？」
→ 必加標題 + 麵包屑

### 紅旗 4：使用者問「下一步做什麼？」
→ 必加 CTA + 引導

### 紅旗 5：手機拇指碰不到主要按鈕
→ 必改版面，把重要按鈕移到下半部

### 紅旗 6：一頁內容塞得很滿
→ 必砍掉至少 30% 內容，或分頁

### 紅旗 7：配色刺眼
→ 必用 `--rise` / `--fall` 系列，不用純紅純綠

### 紅旗 8：動畫讓人頭暈
→ 必放慢速度，或拿掉

### 紅旗 9：失敗訊息讓人焦慮
→ 必改成溫和文案 + 給解法

### 紅旗 10：整個系統感覺是「機器做的」
→ 必加呱呱 + 人味文案

## 💡 Claude Code 做完一頁的最後 3 個問題

Claude Code 交付任何頁面前，問自己這 3 個問題：

### 問題 1：「如果 Vincent 早上爆肝起床，打開這頁會舒服嗎？」
- 顏色不會刺眼？
- 資訊量不會過載？
- 3 秒內能找到重點？

### 問題 2：「如果 Vincent 單手拿手機在捷運上看，能用嗎？」
- 拇指能按到主要按鈕？
- 關鍵資訊在第 1 屏？
- 滑動順暢？

### 問題 3：「這頁讓 Vincent 想關掉，還是想多看？」
- 有「引人繼續探索」的設計？
- 有呱呱的存在感？
- 有讓人微笑的細節？

### 如果 3 題有任何一題答不出「舒服 / 能用 / 想多看」→ **回去改**，不要交給 Vincent。

## 🎁 給 Claude Code 的心法總結

### 心法 1：**程式碼要對，感覺更要對**
功能 100% 正確 + 感覺不對 = 使用者不會用 = 失敗
功能 90% 正確 + 感覺很好 = 使用者愛用 = 成功

### 心法 2：**留白比塞滿更需要勇氣**
每次想「這個也加上去」時，先問「沒有會怎樣」。
沒差 → 不要加。

### 心法 3：**抄好的比發明新的安全**
你不確定怎麼做時，**抄**：
- Linear.app（UI 乾淨）
- Notion（互動細膩）
- Arc Browser（小細節有人味）
- Things 3（動畫優雅）

### 心法 4：**問呱呱怎麼想**
遇到兩難的設計決定，問自己：
> 「呱呱（一隻慢條斯理、不急不徐的老鴨）會怎麼做？」

答案通常是：**更簡單、更慢、更安靜**的那個。

### 心法 5：**做完自己用 5 分鐘**
寫完程式碼 → 自己打開來用 5 分鐘 → 問自己：
- 我會想把這個推薦給朋友嗎？
- 如果不會，為什麼？
